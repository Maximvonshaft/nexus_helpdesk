## Problem

Process health and database connectivity cannot prove that Nexus is ready for an OSR operating mode. A release profile must declare every governed capability as Required, Optional or Forbidden, and one deterministic evaluator must fail closed when evidence is missing, degraded, failed or unexpectedly enabled.

The historical #578 branch mixed this core contract with collectors, API wiring, Settings changes and deployment rollout. Those broader paths are now active in other Work Items or depend on #546/#567. This slice isolates the reusable profile authority so later collectors cannot invent conflicting readiness semantics.

## Scope

### Included

- versioned profile and evidence schemas;
- fixed capability registry;
- complete declarations for `development`, `shadow`, `pilot` and `full_osr`;
- deterministic evaluation with `ready`, `degraded` and `not_ready` precedence;
- bounded low-cardinality reason codes generated by the contract;
- deterministic safe configuration SHA-256 with key-aware redaction;
- strict tests, CI gate and engineering documentation.

### Excluded

- Settings or environment parsing;
- `/healthz`, `/readyz` or Admin APIs;
- Worker, queue, Tracking, Knowledge, Tenant, Provider, Dispatch, recovery or resilience collectors;
- deployment templates and runtime activation;
- database, migration, Provider, outbound or production mutation.

## Contract

### Profile schema

Schema ID: `nexus.osr.release-profile.v1`

Each profile declares every capability exactly once as:

- `required` — missing, disabled or failed is `not_ready`; degraded is `degraded`;
- `optional` — missing or disabled is acceptable; degraded or failed is `degraded`;
- `forbidden` — any enabled/ready/degraded/failed evidence is `not_ready`; missing or disabled is acceptable.

Unknown profiles, unknown capabilities, incomplete declarations and invalid states fail closed.

### Capability registry

The fixed v1 registry covers:

- database and migration identity;
- storage and runtime signing;
- Tenant authority;
- Tracking Truth and Knowledge readiness;
- escalation policy;
- Worker heartbeat and successful progress;
- queue health;
- Provider runtime and canary authority;
- Dispatch execution and acknowledgement;
- external writes;
- observability;
- recovery and resilience;
- AI Runtime contract, RAG v2, RAG sync freshness, deployment identity and Voice runtime.

This registry is intentionally broader than currently implemented collectors. A required capability with no accepted collector remains missing and therefore not ready.

### Profiles

- `development`: core local health required; risky external authority forbidden; most operational capabilities optional.
- `shadow`: read-only truth, worker/progress, queue, observability and AI contract evidence required; external writes forbidden.
- `pilot`: first-class Tenant, Provider canary, Dispatch acknowledgement, recovery/resilience and external writes required.
- `full_osr`: every capability required.

### Evidence

The evaluator accepts only fixed capability states. No raw free-text detail is accepted. Results contain only profile/schema/status and sorted fixed reason codes.

### Configuration fingerprint

The helper returns only a lowercase SHA-256 digest. It:

- sorts mappings deterministically;
- limits depth, mapping entry count, sequence length, key length, string length and numeric magnitude;
- splits snake, kebab, space, acronym and camel-case keys into segments;
- redacts values for terminal secret/password/authorization/credential/cookie/payload/token keys and API/private/access/signing/secret-key pairs;
- treats plural token-count keys such as `max_tokens`, `input_tokens` and `output_tokens` as non-secret configuration so their changes remain visible;
- accepts only bounded JSON-like scalar/container values;
- validates the complete shape before applying any redaction;
- rejects unsupported, non-finite, oversized, over-depth or otherwise ambiguous input.

Secret value changes do not alter the digest because values are redacted; non-secret configuration changes do. Invalid values under a sensitive key still fail closed because validation precedes redaction.

## Status precedence

`not_ready` overrides `degraded`, which overrides `ready`. Reason codes are unique, sorted and generated from fixed capability/state/requirement combinations.

## Security and privacy

The core contract performs no I/O and emits no supplied configuration values. It cannot contact Providers, inspect customer data or mutate runtime state. Configuration fingerprints are not proofs of secrecy; they are bounded identity digests.

## Rollback

Revert the additive module, tests, Workflow and documentation. No runtime, schema, data or external cleanup is required.
